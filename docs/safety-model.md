# 安全模型

执行器会修改用户收藏夹关系，因此首版设计优先保守、可暂停、可恢复。

## 本地运行

- 脚本只在 Bilibili 收藏夹页面内运行。
- 请求使用当前浏览器登录态。
- 不上传 Cookie、SESSDATA、csrf、收藏夹列表或任务包到第三方服务器。
- 状态保存在浏览器 `localStorage`，用户可导出 JSON 备份。

## move 两阶段

`move` 不是一次性删除再添加，而是：

1. 添加到目标收藏夹。
2. 等待随机间隔。
3. 从来源收藏夹移除。

如果在第 1 步后中断，任务会进入 `added_to_target`，下次执行只继续移除来源夹，避免重复添加。

## 节奏控制

执行器包含随机等待和批间休息：

- 普通写操作间隔：约 3-8 秒。
- cleanup 间隔：约 3-7 秒。
- 删除收藏夹间隔：约 5-10 秒。
- 批间休息：几十秒级。
- 长休息：大批量执行时自动触发。

这些间隔不能保证避免风控，只是降低连续写请求风险。

## 熔断条件

遇到以下情况立即停止，不自动重试：

- HTTP `403`
- HTTP `412`
- 登录失效，例如 Bilibili API `-101`
- csrf 缺失
- 写接口返回非成功码
- 验证码、风控、权限异常

挂起后请先导出状态，再人工检查登录、验证码、账号风控或网络状态。

## 备份建议

- 导入任务包后先备份。
- 创建目标夹后再备份。
- 大批量执行中定期备份。
- 删除收藏夹前单独备份。

备份 JSON 可重新导入执行器，继续之前状态。
